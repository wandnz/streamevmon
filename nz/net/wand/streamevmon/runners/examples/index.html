  •  package runners

    Contains entrypoints that run Flink pipelines of various configurations.

    Contains entrypoints that run Flink pipelines of various configurations.

    YamlDagRunner is the default entrypoint. It will run every supported detector according to the program config.

    runners.detectors contains runners for individual detectors, which are less configurable than the default entrypoint. Code changes will be needed if you want to change the data source, for example.

    runners.examples contains runners which demonstrate potentially interesting applications. For example, the MetricGatherer demonstrates gathering some statistic summaries over a tumbling window of measurements.

    This package contains the main entrypoint for this program.

    This package contains the main entrypoint for this program. It reads a desired flow configuration from a YAML file, then constructs and executes that configuration as a Flink pipeline.

    An arbitrary number of sources, detectors, and sinks are supported. They are represented as a directed acyclic graph (DAG). See CONFIGURING_FLOWS.md for details on how to configure this package.

  1.  object AmpMeasurementsToCsv

    Shows saving measurements that implement CsvOutputable to a file using the StreamingFileSink.

    Shows saving measurements that implement CsvOutputable to a file using the StreamingFileSink.

    This example is more convoluted than it needs to be in order to get around restrictions of the file sink in regards to checkpointing and finite streams. If a similar layout was used for an infinite stream, it would be much shorter.

    The StreamingFileSink writes to temporary files, and will solidify them when checkpoints occur if certain conditions are fulfilled. In this example, we use a processing time condition that will only roll a file every 5 seconds. This means that all our example files can be put into one output file.

  2.  object EsmondHistoryWriter extends Logging

    Tools for writing historical Esmond measurements to a local archive.

    Tools for writing historical Esmond measurements to a local archive.

    This doesn't write anything to disk until all the measurements are queried, which can cause very high memory usage when archiving longer durations. It is recommended to provide a higher-than-default -Xmx argument to the JVM.

    The data is written in four formats: Two JSON, two Java-Serialization. Each of these are written both as a large single blob, and as many files: one file for each stream, placed into sensible folders.

  6.  object LatencyTSAllDetectors

    Runs all detectors against each series in the Latency TS I dataset.

    Runs all detectors against each series in the Latency TS I dataset.

    Each file is run separately so that we can more explicitly put the results in the correct files. This could be done with a single Flink pipeline, but using the StreamingFileSink with finite sources is a pain, and naming the files how they should be is a lot more verbose. This is a simpler and nicer solution that just adds a bit of extra overhead to create the pipeline for each file.
